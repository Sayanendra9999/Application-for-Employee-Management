# Admin → HR Employee Onboarding Workflow — Walkthrough

## Overview

This enhancement introduces a **structured onboarding pipeline** between the Admin and HR modules. Previously, when Admin created a new user, an employee record was auto-generated with only a code (`EMP####`), leaving all HR-specific fields (department, designation, salary, bank details, PAN) empty. These employees appeared as "Unassigned" throughout the system but there was no dedicated workflow for HR to act upon them.

The changes below create a complete **Admin → HR → Employee** activation flow without modifying the database schema, without changing existing APIs, and while fully preserving the current role-based access system.

---

## Core Changes

### 1. HR Services Layer (`app/hr/services.py`)

**New functions added at the end of the file:**

| Function | Purpose |
|---|---|
| `is_employee_profile_complete(emp)` | Checks if an employee has all 5 required fields filled: department, designation, salary, bank account, PAN |
| `get_missing_fields(emp)` | Returns a list of human-readable field names that are still missing |
| `get_unassigned_employees()` | Queries all employees and filters (in-memory) for incomplete profiles |
| `get_unassigned_count()` | Quick count wrapper used by dashboard stats |
| `complete_employee_profile(...)` | Validates and applies all required fields to an employee record in a single operation |

**Design decision:** Profile completeness is evaluated in-memory rather than via SQL filters. This avoids schema changes (no new `status` column) while keeping the definition of "complete" purely business-logic-driven and easily adjustable.

### 2. HR Routes (`app/hr/routes.py`)

**Three new endpoints added:**

| Route | Method | Description |
|---|---|---|
| `/hr/employees/unassigned` | GET | Dedicated onboarding queue page listing all incomplete employee profiles with their missing fields |
| `/hr/employees/<id>/complete-profile` | GET, POST | Focused form pre-populated with employee data, highlighting which fields need filling. On submit, validates all required fields, updates the record, re-initializes leave balances, and logs an audit event |
| `/hr/api/employee/<id>/profile-status` | GET | JSON API returning profile completion status and any missing fields |

**Legacy route removed:**

- `POST /hr/employees/add` — **Removed.** This was a legacy route that let HR independently create employee records by linking to user accounts. Since Admin now auto-creates employee records during user creation, this route was redundant and could lead to duplicate entry points. Employee creation is now exclusively Admin's responsibility; HR's role is to *complete* profiles, not *create* them.

**Existing routes modified:**

- `GET /hr/employees` — Now accepts `?status=unassigned` or `?status=assigned` query parameter for inline filtering. Passes `is_profile_complete` function to the template for per-row status rendering. Also passes `unassigned_count` for the queue badge.

- `GET /hr/` (dashboard) — Now computes and passes `unassigned_count` to the template.

### 3. HR Templates

#### New: `hr/unassigned_employees.html`
- Displays a table of all employees with incomplete profiles
- Shows per-employee missing field badges (e.g., "Department", "Salary")
- Provides a prominent **"Complete Profile"** action button per row
- Shows an "All Caught Up" success state when the queue is empty
- Includes an informational banner explaining the Admin → HR workflow

#### New: `hr/complete_profile.html`
- Focused onboarding form for completing an employee's profile
- Displays employee info banner with avatar, name, code, and email
- Shows missing field warnings inline next to each form field
- Organized into sections: Assignment Details and Financial Details
- Includes dynamic department → designation cascade (reuses existing API)
- Employee code field is read-only (auto-generated by Admin)
- Submit button styled as "Complete Profile & Activate"

#### Updated: `hr/dashboard.html`
- Added an **alert banner** when unassigned employees exist, showing the count and a "Review Now" button
- Added **"Onboarding Queue"** as the first quick action link with a warning badge

#### Updated: `hr/employees.html`
- **Removed** the "Add Employee" button (creation is now Admin-only)
- Added a **Status column** showing "Complete" (green) or "Incomplete" (amber) per employee
- Added a **status filter dropdown** (All / Unassigned / Assigned) in the filter bar
- Incomplete employees show a **"Complete Profile"** action button instead of "Edit"
- Added an **"Onboarding Queue"** shortcut button with badge count in the header

### 4. Admin Module Updates

#### `admin/routes.py`
- Dashboard route now computes `unassigned_employees` count via `hr_services.get_unassigned_count()` and passes it to the template
- Users route now passes `is_profile_complete` function to the template

#### `admin/dashboard.html`
- Added an **alert banner** when unassigned employees exist, showing the count and an "HR Action Required" badge
- Gives admins immediate visibility into the onboarding pipeline they feed into

#### `admin/users.html`
- Added an **"HR Status"** column to the users table
- Shows "Onboarded" (green) or "Pending HR" (amber) per user based on their employee profile completeness

### 5. Employee Module — Activation Gate

#### `employee/routes.py`
- Dashboard route now computes `profile_complete` flag via `hr_services.is_employee_profile_complete()` and passes it to the template

#### `employee/dashboard.html`
- When profile is incomplete, shows a **"Profile Setup in Progress"** notice with:
  - Current employee code
  - Department and designation status (showing "Awaiting assignment" in red for missing fields)
  - Informational message about HR completing the setup
- **Full dashboard modules are hidden** behind the `profile_complete` gate
- When profile is complete, the full Employee Space renders normally with all modules

## Data Flow Summary

```
Admin creates user
    ↓
Employee record auto-created with emp_code only
(department_id=NULL, designation_id=NULL, salary=0, bank_account='', pan_number='')
    ↓
Employee appears as "Unassigned" in:
  - HR Dashboard (alert banner with count)
  - HR Employees list (status = "Incomplete")
  - HR Onboarding Queue (dedicated page)
  - Admin Dashboard (alert banner)
  - Admin Users list (HR Status = "Pending HR")
    ↓
HR uses "Complete Profile" form to fill:
  - Department, Designation, Salary, Bank Account, PAN
    ↓
On submit:
  - Employee record updated (single source of truth)
  - Leave balances re-initialized
  - Audit log entry created (COMPLETE_PROFILE)
  - Flash success message
    ↓
Employee removed from "Unassigned" category
Changes immediately reflected across:
  - HR Employees list (status = "Complete")
  - Admin Users list (HR Status = "Onboarded")
  - Admin Dashboard (count decremented)
  - HR Dashboard (count decremented or alert hidden)
    ↓
Employee Space fully activated
  - Full dashboard with all modules visible
  - Attendance, Leaves, Tasks, Documents, etc.
```

## Constraints Honored

| Constraint | How |
|---|---|
| No database schema changes | All logic uses existing `employees` table columns; completeness is derived from NULL/empty checks |
| Use existing APIs for updates | Reuses existing `EmployeeForm`, existing `services.get_departments_for_dropdown()`, `services.get_designations_for_dropdown()`, `services.initialize_leave_balances()`, etc. |
| Maintain role-based access | All new routes use `@module_required('hr')` decorator. Employee dashboard gating is template-level only (route still accessible). Admin routes use `@admin_required`. |
| Single source of truth | The `employees` table remains the single record. No duplicate records or shadow tables. All modules read from the same row. |
| Sync across modules | Because all views query the same `Employee` model, changes are reflected immediately everywhere upon `db.session.commit()`. |

## Architectural Cleanup: "Add Employee" Removed from HR

The legacy `add_employee` route in HR was removed because it violated the clean separation of concerns established by the onboarding pipeline:

| Concern | Before | After |
|---|---|---|
| **Who creates users?** | Admin | Admin (unchanged) |
| **Who creates employee records?** | Admin auto-creates AND HR could independently create | Admin auto-creates only |
| **Who completes employee details?** | HR (via edit or add) | HR (via dedicated Complete Profile form) |
| **Entry points for employee creation** | 2 (Admin + HR) | 1 (Admin only) |

This eliminates the risk of duplicate entry points, confusion about ownership, and the empty `users_without_profile` pool that `add_employee` relied on (since Admin auto-creates employee records, there are never users without profiles).
